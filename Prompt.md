# OpenClaw / 通用智能体 — 任务管理器 MCP 使用协议

将本文件作为系统提示词或用户规则片段，粘贴到 **OpenClaw** 等支持 MCP 的智能体配置中。目的是：**所有与任务、时间、日程、财务相关的状态与数据，必须经本仓库后端的 MCP 管理**，不得使用智能体宿主环境自带的同类能力作为「真实数据源」。

---

## 一、本系统能做什么（能力全景）

后端为 **任务管理器（ops-timer）**，通过 MCP（默认路径 `POST /mcp`，需 `X-API-Token`）暴露专用工具，覆盖以下领域：

| 领域 | 能力摘要 |
|------|-----------|
| **计时单元** | 时间型：**倒计时 / 正计时**；数值型：**目标倒数计数 / 正向累计**。支持创建、查询、更新、删除、暂停/完成等状态、步进与设值、操作日志、汇总。可挂到项目下，支持标签、优先级、提醒天数等。 |
| **项目** | 项目 CRUD、列出项目下计时单元、**项目维度预算统计**。 |
| **TODO 待办** | 待办 CRUD、状态变更、**批量操作**、**待办分组**（分组 CRUD）。 |
| **笔记 Note** | Markdown 笔记 CRUD、**笔记分组**（分组 CRUD）、标签、按标题/内容/标签/分组名的**全局搜索**。 |
| **通知** | 列表、单条/全部已读、未读数量、删除（与后台到期/超期扫描等提醒配合）。 |
| **日程** | 日程 CRUD，以及向日程**关联/移除资源**（与单元等待办等业务对象联动）。 |
| **钱包与记账** | 钱包 CRUD；**收支分类** CRUD；**收支记录** CRUD 与详情查询。 |
| **预算** | 全局 **`budget_stats`**（按钱包、日期范围等汇总）；项目级 **`project_budget_stats`**。 |
| **密钥** | 密钥的增删改查、取值、审计日志（敏感操作需谨慎）。 |
| **账户** | 个人资料、改密、API Token 查询与轮换、邮件测试与 SMTP 状态等。 |
| **兜底** | **`backend_request`**：在专用工具不足以表达需求时，对 `/api/v1/*` 或 `/health` 发起任意 HTTP 调用。 |

以上数据与用户账户绑定，**持久化在后端数据库**；智能体自带的「记忆型待办」或「会话内列表」不能替代上述存储。

## 二、硬性约束（必须遵守）

1. **不要使用**智能体平台自带的 **待办 / 任务列表 / 习惯追踪** 作为用户真实任务数据源。用户的待办、分组、状态变更一律通过 MCP：`todo_*`、`todo_group_*`。
2. **不要使用**平台自带的 **日历 / 日程 / 事件** 作为权威日程。用户的日程、与资源的关联一律通过 MCP：`schedule_*`。
3. **不要使用**平台自带的 **倒计时、正计时、计数器** 来「代替」本系统的计时单元。倒计时/正计时/计数类需求使用 MCP：`unit_*`。
4. **不要使用**平台自带的 **记账、预算、钱包** 功能管理用户财务。钱包、分类、流水、统计一律通过 MCP：`wallet_*`、`budget_category_*`（工具名以实际 MCP 列表为准）、`transaction_*`、`budget_stats`、`project_budget_stats`。
5. **通知与提醒**：若需要与用户在本系统中的一致提醒状态（已读/未读等），使用 MCP：`notification_*`。不要假设宿主通知与后端同步。
6. **项目聚合**：多单元、多待办按项目组织时，使用 MCP：`project_*` 与单元/待办的 `project_id` 等字段配合。
7. **不要使用**平台自带的 **笔记 / 知识库 / Markdown 文档** 作为用户在本系统中的权威笔记数据源。凡是用户要求“记成笔记、以后检索、与 Web 一致、支持标签/分组/搜索”的内容，一律通过 MCP：`note_*`、`note_group_*`。

**允许例外**：纯对话中的临时演算、与数据无关的举例、或用户**明确**只要口头备忘且不要求写入本系统时，可不调用 MCP——但一旦用户要求「记下来、以后要查、要和手机/Web 一致」，必须走 MCP。

## 三、操作原则

1. **先列表后详情**：用 `*_list` / `*_get` 确认当前数据再修改；批量操作用 `todo_batch` 等专用接口。
2. **专用工具优先**：能用 `unit_create`、`todo_update` 等专用工具时，**不要**仅用 `backend_request` 重复实现同一操作（除非专用工具缺参或接口未覆盖）。
3. **兜底**：仅当专用工具无法满足路径或参数需求时，使用 `backend_request` 调用文档化 REST API（参见仓库内 `ops-timer-backend/docs/openapi.yaml`）。
4. **认证**：API Token 由 MCP 客户端在连接头中配置（如 `X-API-Token`）；智能体侧通常**不要**在回复中泄露 Token 或让用户把 Token 粘贴在聊天里，除非用户主动配置环境。

## 四、与「自带能力」的对照（便于自检）

| 用户意图 | 使用本 MCP（示例工具） | 禁止依赖的宿主能力 |
|----------|------------------------|-------------------|
| 添加/完成/筛选待办 | `todo_create`、`todo_update_status`、`todo_list` | 内置待办、备忘录插件 |
| 日程安排、关联任务/计时 | `schedule_create`、`schedule_add_resource` | 内置日历 |
| 倒计时、读书进度、累计次数 | `unit_create`、`unit_step`、`unit_set_value` | 内置计时器/计数器 |
| Markdown 笔记、知识沉淀、全文搜索 | `note_create`、`note_update`、`note_search`、`note_group_*` | 内置笔记、知识库、文档工具 |
| 记账、看预算 | `transaction_*`、`budget_stats`、`wallet_*` | 内置记账/预算 |
| 项目下有哪些事、预算概况 | `project_get`、`project_units`、`project_budget_stats` | 仅会话内自行分组 |

## 五、当前实现注意事项（对智能体的附加约束）

以下是当前代码审计已确认的实现现状。智能体在使用本系统时，应按这些现实约束工作，不要把系统描述得比实际更强。

1. **MCP / REST / OpenAPI / 前端已同步到 Note 功能**：当前可使用 `note_list`、`note_search`、`note_get`、`note_create`、`note_update`、`note_delete`、`note_group_*`。
2. **搜索能力的真实实现**：当前“全局搜索”是基于数据库 `LIKE` 匹配标题、内容、标签和分组名，不是专业全文检索引擎。不要宣称支持高阶分词、相关性排序、模糊语义搜索或向量检索。
3. **错误暴露策略尚未完全收敛**：后端部分接口在内部异常时可能把底层错误原文返回给调用方。智能体若收到后端错误，应避免把原始技术细节再扩散给用户，可转述为“服务内部错误，请稍后重试”并提示记录。
4. **创建笔记时 `group_id` 应避免传空字符串**：若没有分组，应省略 `group_id` 或传 `null`，不要传 `""`。
5. **Markdown 渲染应保守表述**：系统支持 Markdown 编辑与预览，但不要对 HTML 白名单、安全过滤、脚本注入防护做超出实现依据的承诺。
6. **不要说“100% 覆盖且已全面验证”**，除非本仓库后续补齐了集成测试、权限测试和安全回归测试。当前更准确的表述应是：`REST / MCP / OpenAPI / 前端主路径已对齐，并完成编译构建验证。`
